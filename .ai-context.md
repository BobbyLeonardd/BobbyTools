# BobbyTools: Project Memory & Architecture Context

**ATTENTION FUTURE AI ASSISTANTS:**
Read this file to understand the architecture, purpose, and design philosophy of BobbyTools before making any code changes.

## 🚀 What is BobbyTools?
BobbyTools is a "lazy genius" developer tool designed to eliminate the pain of managing `.env` files for developers who use multiple AI CLIs (like `opencode`, `aider`, `claude-code`) and juggle multiple API keys (free tiers, client keys, etc.). 

It operates in three main modes:
1. **Classic Launcher (`bobby`)**: An interactive CLI menu that lets the user select a Provider, Account (API Key), and Model, and then launches their target CLI tool with the credentials injected directly into the RAM/environment.
2. **Local AI Router / 9Router Mode (`bobby serve`)**: A lightweight HTTP proxy server running locally (default: `127.0.0.1:13337`). It acts as a universal AI gateway. Users point their CLIs to this local server, and BobbyTools automatically forwards requests to the real APIs.
3. **Web Dashboard Daemon (`bobby serve-bg`)**: The Local AI Router runs detached in the background. Users can manage their configurations via a Vanilla HTML glassmorphism web interface at `http://127.0.0.1:13337`. Features real-time config sync (via `configHash` polling) and live Router Activity Logs.

## 🔥 Killer Feature: Auto-Rotate (The 429 Handler) & Smart Fallback
The absolute core value of the Local Router is **Auto-Rotation** and **Round-Robin Load Balancing**. 
When a user has multiple accounts (keys) for a single provider (e.g., 5 Groq keys), the router will track them and split the traffic round-robin across all active accounts. If a request hits a `429 Too Many Requests` (or 401/402) limit, the router catches the error, silently swaps to the next available API key, and retries the request in the background. The CLI tool receiving the final response never knows an error occurred.

Even better: **Universal Cross-Provider Fallback**. If ALL accounts for a specific provider (e.g., Groq) are limited, the router searches across the user's *entire* BobbyTools configuration for *another* provider (e.g., OpenRouter) that has active accounts and supports the *exact same model* (e.g., `llama3-70b-8192`). It will seamlessly switch providers mid-request without breaking the client!

## 🏗️ Technical Stack & Architecture
- **Language**: Node.js (ES Modules, `>=18.0.0` required).
- **No Heavy Dependencies**: We rely on native modules (`http`, global `fetch`, `Buffer`) for the router. The only external libs are `chalk` (styling), `@inquirer/prompts` (UI), and `commander`.
- **Config Storage**: All user data (Providers, Accounts) is stored in a plain text JSON file at `~/.bobbytools/config.json`.
- **Real-Time Config Reading**: The router *does not* hold state or a separate database for providers. Every HTTP request dynamically calls `getConfig()` to read `config.json`. This means users can add/delete accounts in the CLI menu, and the running router instantly recognizes the changes without needing a restart.
- **Routing Logic (`src/server.js`)**:
  - `GET /`: Serves the `dashboard.html` Single-Page Application.
  - `GET /api/config`, `POST /api/config`, `POST /api/shutdown`, `GET /api/ping`, `GET /api/logs`: Endpoints for the Web Dashboard CRUD, daemon lifecycle management, and live activity logs.
  - `GET /v1/models`: Aggregates models from ALL providers defined in `config.json` and returns them in OpenAI's standard format.
  - `POST /v1/chat/completions`: Extracts the target provider and model from the request body's `model` string (format: `providerId/modelName`). Looks up the provider's base URL and the active API key, rewrites the request, and proxies the stream/JSON back to the client.

## 🧠 Design Philosophy (The "Lazy Genius" Persona)
- **Zero Configuration Boilerplate**: The tool must be plug-and-play.
- **Minimal Code**: Avoid large frameworks (no Express, no Axios). Use native `http` and `fetch` to keep execution fast and binary sizes small.
- **Tone**: The README and terminal UI are written in an arrogant-but-helpful "lazy senior developer" persona (Indonesian tech slang: "gue", "lo", "tuyul"). Keep this tone if rewriting documentation.

## 🤯 The "Inception / Loopback" Architecture Hack
Users can exploit BobbyTools to get **both** the interactive CLI menu and the Auto-Rotate feature without modifying their external tools' configuration:
1. They run `bobby serve` in Terminal 1 (`127.0.0.1:13337/v1`).
2. They run `bobby` in Terminal 2, create a Custom Provider (e.g. `LocalRouter`), set the Base URL to the router (`127.0.0.1:13337/v1`), and set it to fetch models dynamically.
3. The interactive menu fetches models from the running router, presenting a combined dropdown of ALL models across ALL providers.
4. When they start a session, `bobby` launches the external CLI pointing to `127.0.0.1:13337/v1`. The router receives it, peels off the `LocalRouter/` prefix, and successfully forwards it to the real provider with Auto-Rotate enabled.
**CRITICAL PATCH:** To prevent an infinite fractal loop during model fetching (where the router fetches models from itself recursively), the `GET /v1/models` endpoint explicitly skips any provider containing `127.0.0.1` or `localhost` in its URL. **Do not remove this check.**

## ⚠️ Known Quirks & Fixes
- `@inquirer/prompts` modifies `process.stdin` and pauses it. When building the `bobby serve` interactive UI (which listens for 'b' or 'q' to go back), we had to explicitly call `process.stdin.resume()` and handle raw mode toggling to regain control of the terminal input.
- **Windows vs Mac/Linux Env Vars**: The documentation explicitly teaches users the difference between `$env:VAR="value"` (PowerShell) and `export VAR="value"` (Bash) because many users fail to set the router's URL properly in their terminals.
- **Header Case-Sensitivity (v2.1.2)**: Node.js converts `req.headers` keys to lowercase (`authorization`). The router previously tried to inject API keys using `Authorization`, resulting in both headers being sent and Cloudflare/providers returning `401 Unauthorized`. It is now fixed to use lowercase `authorization`.
- **ANSI Escape Fix (v2.1.2)**: Fixed literal `\x1b` being logged instead of actual ANSI escape codes for clearing terminal lines.
- **UI Dim Function Quirk (v2.1.3)**: The `dim()` function in `ui.js` executes `console.log` immediately and returns `undefined`. It must not be used inside template literals for string formatting (e.g., in Inquirer prompts). Use `chalk.gray()` instead.
- **Inquirer Scroll vs Loop Bug (v2.2.0)**: Inquirer's `select` prompt defaults to `pageSize: 7`. For menus like CLI Tools selection, adding too many options causes the list to scroll, hiding the "Back" button at the bottom. We resolved this by explicitly defining `pageSize: 15` across all menus while retaining the default `loop: true` to allow cursor wrap-around from top to bottom.
- **Manual Model Caching Bug (v2.2.1)**: Previously, manually entered models (for providers without endpoints) were saved to disk but didn't update the in-memory `provider.models` array, forcing users to re-type the model if they stayed in the same CLI session. We fixed this in `src/models.js` by always re-fetching the fresh provider from `getConfig()` and updating the in-memory array (`provider.models.push`).
- **CLI Back Button Navigation UX (v3.0.0)**: In CLI mode, canceling manual input (e.g. typing '<' in manual model selection or custom CLI command) would drop the user all the way back to the Account Selection step. Fixed by converting the manual inputs into recursive functions that re-call the parent menu upon cancellation.
- **Custom Provider Crash (v3.0.0)**: Adding a Custom Provider resulted in `models` array being `undefined` rather than `[]`, causing a crash when saving manual models later. Initialized `models: []` uniformly across CLI and Web Dashboard.
- **CLI Global Target Overhaul (v3.1.0)**: Previously, CLI commands were globally managed in a list (`cliTools`) and globally defaulted in settings, leading to UX friction. In v3.1.0, global tools were entirely removed. Now, a `targetCliCommand` is stored directly on the `provider` level (and optionally inputted when adding a provider). 
- **Web UI "Set Target" Removal (v3.1.0)**: The Web UI historically featured a "Set Target" button to designate the active CLI. This was completely removed because the Local Gateway (port 13337) inherently handles multi-provider proxying dynamically via the `providerId/modelName` format (Smart Load Balancing). It was a legacy feature that confused users.
- **Responsive & Interactive UI (v3.1.0)**: The Web Dashboard underwent a massive overhaul to a fully fluid, zoom-agnostic, responsive glassmorphism design. All modals feature `overflow-y` and click-outside-to-close behavior to ensure users are never blocked on small screens. A real-time Search Bar was also added to the Provider list to manage large datasets easily.
- **In-App Tutorials & Docs (v3.1.0)**: Added highly detailed, "lazy developer" persona in-app tutorials for both the CLI Menu (`src/index.js`) and the Web Dashboard (`src/dashboard.html`). Also entirely rewrote `README.md` to be extremely clear for new users avoiding env-var hell.
